# Preference Memory

Use this area for durable user preferences, risk posture, recurring cautions,
and taste that should guide future work.

## How To Use

- Use dated headings for new or changed preferences.
- Keep entries concrete and searchable.
- Promote repeated procedure into runbooks or skills.
- Mark superseded preferences explicitly.
- Do not store secrets or credential values.

## Reusable Preference Examples

- Preserve historical memory while keeping current state fast to find.
- Keep notes human-readable as plain Markdown and friendly to grep, git, and
  Markdown vault tools.
- Prefer a wiki-plus-ledger durable notes model.
- Tests must clean up after themselves.
- Keep credentials scoped and revocable.
- Prefer GitHub Apps over personal access tokens for GitHub automation.
- Use Linear as the canonical backlog while keeping local execution inspectable.
- Prefer boring local infrastructure: systemd, tmux, SSH, Markdown, and plain
  files.
